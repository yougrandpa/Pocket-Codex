## 3. Bug 清单

### Bug 1：登录页预填默认管理员账号密码，可一键登录
- 严重级别：`S1 - 严重`
- 复现步骤：
  1. 打开 `http://localhost:3000/` 未登录页。
  2. 观察用户名、密码输入框，默认已填入 `admin` / `admin123`。
  3. 直接点击“登录”。
- 期望结果：
  - 登录表单默认应为空；
  - 不应存在可直接使用的默认管理员凭据。
- 实际结果：
  - 页面预填且可直接登录成功。
- 影响：高风险账号泄露面，极易被弱口令利用。

### Bug 2：手机端登录未要求“电脑端授权确认”
- 严重级别：`S1 - 严重`
- 复现步骤：
  1. 切换移动端视口（`390 x 844`）。
  2. 直接使用账号密码登录。
  3. 全程无电脑端二次确认/授权。
- 期望结果：
  - 手机登录必须经已登录电脑端确认（配对授权、一次性挑战、可撤销）。
- 实际结果：
  - 手机端可独立完成密码登录，未触发电脑授权链路。
- 影响：与“手机登录必须由电脑授权”的安全要求不符。

### Bug 3：审计日志分页不可达（总数固定 20）
- 严重级别：`S2 - 高`
- 复现步骤：
  1. 登录后执行多次会触发审计的操作（创建任务、follow-up、退出再登录）。
  2. 查看“审计日志”模块。
  3. 观察分页信息与按钮状态。
- 期望结果：
  - 日志总数应随新增事件增长；
  - 当超过单页上限时应可翻页。
- 实际结果：
  - 页面持续显示“共 20 条日志 / 第 1 / 1”；
  - 下一页按钮始终禁用；
  - 网络请求固定为 `GET /api/v1/tasks/audit/logs?limit=20`。
- 影响：历史审计追溯能力不足，影响排障与合规审计。

### Bug 4：英文模式翻译不完整（占位符残留中文）
- 严重级别：`S3 - 中`
- 复现步骤：
  1. 进入任务详情页。
  2. 点击“English”切换到英文。
  3. 查看“Append instruction”输入框占位符。
- 期望结果：
  - 英文模式下所有文案（含 placeholder）应完整英文。
- 实际结果：
  - 占位符仍显示中文“补充指令...”。
- 影响：多语言一致性与专业感受受损。

### Bug 5：移动端存在横向溢出，页面可左右滑动导致内容截断
- 严重级别：`S3 - 中`
- 复现步骤：
  1. 视口切换为 `390 x 844`。
  2. 浏览首页卡片（尤其“执行器状态/路径信息”区域）。
  3. 观察出现横向滚动与左侧内容被裁切。
- 期望结果：
  - 移动端不应出现横向滚动；内容应在容器内换行/截断。
- 实际结果：
  - 页面出现横向偏移，可左右滑动，文本和模块局部被截断。
- 影响：移动端可读性与可操作性下降。

### Bug 6：follow-up 指令执行结果未按新指令响应

### Bug 7：基础静态资源缺失与表单安全提示

- 影响：细节体验和安全基线完善度不足。

## 4. 体验提升建议
1. **任务流反馈更明确**：在“创建任务/发送消息”后增加进度阶段（已入队、执行中、完成）与可见倒计时，降低用户不确定感。
2. **多语言质量门禁**：建立 i18n 文案完整性检查（含 placeholder、按钮、空态、错误提示），避免语言切换后混杂。
3. **移动端信息分层**：将执行器细节（路径、超时、并发）折叠为“高级信息”，默认展示关键状态和高频操作。
4. **审计可读性增强**：支持按事件类型/任务 ID 筛选，并加“导出 CSV/JSON”，提升排查效率。
5. **follow-up 结果对齐提示**：在 UI 中高亮“本次运行使用的最新用户消息”，便于确认系统是否采用新指令。

## 5. 性能优化建议
1. **任务列表请求分页化**：当前请求为 `GET /api/v1/tasks?limit=200`，建议改为服务端分页（如 20/50 一页）+ 增量加载，减少首屏负载。
2. **审计接口支持游标分页**：避免固定 `limit=20`；改为 cursor/offset 分页并返回真实总数或 `has_more`。
3. **健康检查降频与退避**：`/healthz` 请求频繁，建议在稳定期降低频率并使用指数退避，减少无效网络开销。
4. **日志列表虚拟滚动**：任务详情日志条目较多时启用虚拟列表，降低 DOM 体积与重排开销。
5. **静态资源完善**：补齐 `favicon.ico`，并检查静态资源缓存策略，减少不必要 404 与重复请求。

## 6. 隐私与安全建议（重点）

### 6.1 手机登录必须由电脑授权（重点，建议强制上线）
建议将手机登录改为“电脑授权配对”模式，而非手机单独密码登录：
1. 电脑端（已登录会话）生成**一次性配对二维码**（短时效 `nonce` + `device_challenge`）。
2. 手机端扫码后仅发起“登录申请”，不直接获得会话。
3. 电脑端弹出确认框（显示手机设备名、IP、地区、时间），用户手动“允许/拒绝”。
4. 仅在电脑端确认后，手机端才下发短期访问令牌，并绑定设备指纹。
5. 支持随时撤销设备授权、查看最近授权记录、异常登录告警。

### 6.2 账号与凭据基线
1. 移除任何默认账号密码预填与硬编码弱口令。
2. 首次启动强制管理员设置高强度密码（最小长度、复杂度、泄露口令检测）。
3. 密码输入增加 `autocomplete="current-password"` 与密码管理器兼容策略。

### 6.3 会话与权限控制
1. Token 使用 `HttpOnly + Secure + SameSite=Strict`；访问令牌短时效、刷新令牌轮换。
2. 对移动端默认隐藏敏感运行细节（模型、provider、工作目录、会话 ID 等），按角色按需开放。
3. 对 `task.message.append`、`task.create` 等高敏操作增加风险校验（最近授权校验/二次确认）。

### 6.4 审计与防滥用
1. 审计日志应保留完整链路与不可篡改策略（签名或哈希链）。
2. 登录/消息接口加速率限制、失败锁定与告警。
3. 提供按设备维度的审计视图，便于定位被盗用终端。

---

## 7. 关键结论
- 核心功能可用，但当前版本在安全模型上不满足“手机登录须电脑授权”的关键要求。
- 建议优先级：
  1. **P0**：手机登录改为电脑授权配对流程 + 移除默认凭据预填。
  2. **P1**：修复审计分页（真实总量与翻页能力）。
  3. **P1**：修复移动端横向溢出与 i18n 残留文案。
  4. **P2**：处理 favicon、autocomplete、健康检查降频等质量项。
